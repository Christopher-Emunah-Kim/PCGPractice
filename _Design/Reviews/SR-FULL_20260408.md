## 5. 아키텍처 수준 강점

### 5-1. 데이터 파이프라인 일관성
- CSV -> DataTable -> GDS 캐시 -> Subsystem -> GA/Character 파이프라인이 전체 시스템에 일관 적용
- GDS가 DT 접근의 유일한 창구(Single Source of Truth)로 잘 기능
- 복합 조회(GetSkillExecutionData, GetWeaponSlotEquipData)가 호출자의 DT 의존을 제거

### 5-2. 서브시스템 책임 분리
- GameInstance vs World 수명 구분이 명확 (GDS/RDS/SGS/EQS/LUS/UMS = GI, SMS/Pool = World)
- 각 서브시스템의 단일 책임이 잘 지켜짐 (GDS=조회, RDS=런타임 상태, SGS=영속, EQS=장비, LUS=레벨업)
- 서브시스템 간 통신이 델리게이트 기반으로 느슨한 결합 유지

### 5-3. GA 계층 구조
- GA_Base (ActivateAbility final + 공통 캐싱) -> GA_ProjectileAttack / GA_SummonBase 분기가 합리적
- SourceObject(URSSkillData) 패턴으로 GA 클래스 재사용성 확보 (1 GA 클래스 = 다수 스킬 데이터)

### 5-4. 풀링 아키텍처
- IPoolableInterface -> OnPoolActivate/Deactivate 계약이 BaseProjectile, EnemyProjectile, BaseSummonObject, EnemyBaseCharacter에 일관 적용
- PoolingSubsystem이 타입 무관 범용 풀로 설계되어 확장성 확보

### 5-5. 로깅 시스템
- 6단계 레벨, 빌드별 컴파일 타임 필터링, 자동 함수명/파일명/라인 포함
- Shipping 빌드에서 Trace/Debug 자동 제거

### 5-6. UI 관리 아키텍처
- 4-레이어(PERSISTENT/PAGE/POPUP/SYSTEM) + ZOrder 자동 계산
- ID 기반 + 클래스 기반 이중 API 호환
- PAGE 히스토리 스택으로 BackPage() 네비게이션 지원

## 7. 전체 평가

| 항목 | 점수 | 근거 |
|------|------|------|
| 아키텍처 | 4/5 | 레이어 분리 명확, 의존 방향 올바름, 순환 의존 없음. GI/World 서브시스템 수명 구분 합리적. 다만 EQS/LUS의 수동 Init/Deinit 패턴이 취약점 |
| 일관성 | 4/5 | GDS 단일 창구, IPoolableInterface 계약, AddDynamic NativeOnInitialized 배치 등 대부분 일관. GA/EQS 간 템플릿 중복, 일부 NativeConstruct 잔류가 감점 요인 |
| 중복 | 3/5 | Enemy 투사체 발사 로직 3곳 중복, 파라미터 필드 15개 중복, LoadRequiredClass 2곳 중복. 플레이어 측은 GA_Base 공통화가 잘 되어 있으나 에너미 측 확장에서 추상화 부재 |
| 기술부채 | 3/5 | ActorPool UPROPERTY(HIGH), SPAWN_OFFSET 하드코딩(4회차 미해결), USTRUCT 기본값 12필드 미설정. 기존 패턴 해결률 ~60%. 신규 시스템(Enemy Expansion) 추가 시 기존 부채 위에 새 부채 적층 |

### 종합: 3.5 / 5

프로젝트의 근간 아키텍처(Data Pipeline, Subsystem 레이어, GA 계층, 풀링)는 견고하다. 이전 SR에서 지적된 switch fall-through, AddDynamic 위치, 풀링 리셋 등은 대부분 교정되어 학습 반영이 진행 중이다. 그러나 하드코딩 수치(4회차), UPROPERTY 누락(3회차), USTRUCT 기본값 미설정 등 데이터 안전성 관련 반복 패턴이 여전히 잔류한다. Enemy Expansion(M7)에서 Ranged/Elite/Boss 3종 확장 시 투사체 발사 로직과 파라미터 선언의 코드 중복이 새로 발생했으며, 이는 다음 Enemy 타입 추가 시 더 심화될 구조적 문제다.
