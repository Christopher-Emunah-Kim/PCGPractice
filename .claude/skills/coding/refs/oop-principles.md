# OOP 설계 원칙 (리뷰 체크 기준)

## GRASP 핵심 원칙
- **정보 전문가**: 필요한 정보를 가장 많이 아는 객체에게 책임을 부여
- **창조자**: 인스턴스를 포함/참조/초기화하는 객체가 생성 책임을 가짐
- **다형성**: 타입별 분기는 조건문이 아닌 다형성으로 해결
- **변경 보호**: 변화가 예상되는 지점을 추상화로 격리
- **간접화**: 직접 의존 대신 중재 객체를 통해 결합도를 낮춤

## 설계 체크포인트
- 객체의 책임 서술에 접속사("그리고", "또한")가 필요하면 책임이 너무 크다
- 일부 필드가 일부 함수에서만 사용되면 클래스 분리를 검토
- 배타적으로 초기화되는 필드가 있으면 분류 가능
- private 함수가 과도하게 많으면 여러 책임을 갖고 있을 수 있다
- 외부 의존성은 의존성 주입으로 처리

## 의존성 관리
- 순환 의존 금지
- 객체가 직접 의존하는 객체하고만 소통 (기차 충돌 금지)
- TDA 원칙: 상태를 묻지 말고 작업을 시켜라 (Getter로 꺼내서 로직 구성 금지)

## 상속 원칙
- IS-A 관계가 아니면 상속하지 않는다
- 코드 중복 방지 목적의 상속 금지 → 합성(Composition)으로 대체
- 상속 계층은 3단계 이하로 유지
- 구현 상속보다 인터페이스 상속 우선

## 객체 생성
- 객체는 생성 즉시 유효한 상태여야 한다 (완전한 객체)
- 생성자에서 초기화를 완료한다
- 복잡한 초기화는 빌더/팩토리 패턴 사용
