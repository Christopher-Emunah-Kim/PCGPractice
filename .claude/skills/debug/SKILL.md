# /debug RUNBOOK
> 역할: 버그/에러 → 체계적 진단 순서 준수 → 원인 특정 → 수정
> 원칙: 진입점 확인 전 내부 진단 금지. git diff가 진단의 시작점.

## DIAGNOSIS_ORDER
⚠️ 이 순서를 반드시 지킨다. 3단계 전에 4단계 진입 절대 금지.
```
1. git diff    → 최근 변경 파악 (범위 먼저 좁히기)
2. 진입점 확인 → 함수가 실제로 호출되는가? (브레이크포인트/로그 요청)
3. 레이어 좁히기 → 어느 레이어에서 깨지는가?
4. 내부 진단  → 원인 특정 (2번 완료 후에만 진입)
5. 수정       → 최소 변경 원칙
6. 회귀 체크  → 인접 기능 영향 없음 확인
```

## STATE_MACHINE
```
INIT ──→ [A] 증상 수집
          └─ [B] git diff 확인 (범위 좁히기)
                └─ [C] 진입점 확인 ← 반드시 2번째
                      └─ [D] 레이어 좁히기
                            └─ [E] 내부 진단 + 수정
                                  └─ [F] 회귀 체크 + 패턴 기록
```

## EXEC

### [A] 증상 수집
```
사용자로부터 확인:
- 기대 동작 vs 실제 동작 (한 문장씩)
- 발생 조건 (항상 / 특정 상황)
- 에러 메시지 있으면 전체 복사 요청
- 마지막으로 정상 동작한 시점
```

### [B] git diff 확인
```bash
git diff HEAD~1 -- Source/
```
→ 최근 변경 파일 목록 파악 → 관련 범위 좁히기
→ 변경 없는 경우: 외부 의존성 / 타이밍 문제 / 엔진 버전 의심

### [C] 진입점 확인
```
⚠️ 이 단계를 건너뛰고 [D]/[E]로 직행 절대 금지.

사용자에게 요청 형식:
  "📌 [DEBUG] 진입점 확인 요청
   [함수명 / 이벤트명]에 KHS_DEBUG 로그를 추가하거나
   브레이크포인트를 설정 후 재실행해주세요.
   - 로그/중단 확인됨 → [D]로 진행
   - 로그/중단 없음   → 호출 자체가 안 됨 → 호출부([상위함수]) 확인 필요"
```

### [D] 레이어 좁히기
```
UE5 레이어 순서 (위에서 아래로 체크):
  Input → PlayerController → Character / Component → Subsystem → GAS (GA / GE / ASC)

각 레이어 경계에서:
  "이 지점까지는 정상인가?" → Yes → 다음 레이어
                             → No  → 이 레이어가 범인
```

### [E] 내부 진단 + 수정
```
원인 특정 후:
- 수정 범위: 최소 변경 원칙 (관련 없는 코드 손대지 않음)
- 버그 수정은 활성 PLAN 없어도 허용
- 수정 후 [D2] 빌드 검증 요청 (/coding과 동일 형식)
```

### [F] 회귀 체크 + 패턴 기록
```
□ 수정 후 원래 동작 정상?
□ 인접 기능 영향 없음?
□ 같은 패턴이 다른 파일에도 있는가? (Grep으로 확인)

반복 패턴 감지 시:
  .claude/memory/senior-reviewer/MEMORY.md 갱신
  형식: pattern: [이름]  count: N  status: RECURRING
```

## RULES
```
- DIAGNOSIS_ORDER 위반 금지 (특히 [C] 전 [E] 진입)
- 수정 범위 최소화 — 버그 수정에 리팩토링 혼재 금지
- 원인 불명 3회 → BLOCKED + 추가 컨텍스트 요청
- 수정 완료 후 SR 권장 (동일 패턴 전파 감지 목적)
- 삭제 전 if/else 분기 전체 영향 확인 필수
```
